# design.md 記載ルール

技術設計と意思決定の記録（ADR）を記載する。上限は **150行**。

## フォーマット

```
## 1. アーキテクチャ概要
システム全体の構成（テキストまたは箇条書き）

## 2. コンポーネント設計

### [コンポーネント名]
- 役割: 〜
- 入力: 〜
- 出力: 〜
- 処理フロー: 〜

## 3. 意思決定の記録（ADR）

### ADR-001: [決定タイトル]
- 決定日: YYYY-MM-DD
- 状態: 採用済み / 廃止 / 検討中
- 背景: なぜこの決定が必要だったか
- 選択肢:
  | 案 | メリット | デメリット |
  |----|----------|------------|
  | A案（採用） | 〜 | 〜 |
  | B案 | 〜 | 〜 |
- 決定理由: なぜA案を選んだか（ユーザーの要望・制約を明記）
- 影響: この決定によって何が制約されるか

## 4. データフロー
処理の流れをテキストで記述

## 5. セキュリティ考慮事項
個人情報・認証・権限管理等

## 6. 非機能要件への対応
パフォーマンス・可用性・監視等
```

## ADRルール
- 技術選定の理由と代替案との比較を**必ず**記録する
- 「あえてこの方式を採用した」理由を明記する（将来のAIが誤って設計を破壊しないための保険）
- 実装中に設計を変更した場合は、その都度 ADR を追記する
