# tasks.md 記載ルール

実装タスクを Phase 別に分割して記載する。

## タスクIDの形式
`TASK-001`, `TASK-002`, ... （3桁ゼロ埋め）

## フォーマット

```
## 実装フェーズ概要
- 総タスク数: X件
- 完了条件: requirements.md の全 REQ-XXX / NFR-XXX が検証済みであること

## Phase 1: 基盤構築

### TASK-001: [タスク名]
- 概要: 何をするか（1文）
- 対応要件: REQ-001, REQ-002
- 依存: なし
- 実装内容:
  1. 〜を作成する
  2. 〜を設定する
- 完了基準: 〜が正常に動作することをテストで確認
- テスト方法: `python test_xxx.py` を実行し、全テストがPASSすること

## Phase 2: コア機能実装

### TASK-002: [タスク名]
...

## Phase 3: 統合テスト・最終確認

### TASK-XXX: エンドツーエンドテスト
- 完了基準: 全 REQ-XXX / NFR-XXX がテスト済みかつPASS

## 優先順位マップ
- High（必須）: TASK-001, TASK-002
- Medium（重要）: TASK-003
- Low（あれば良い）: TASK-004
```

## ルール
- 各タスクは必ず `requirements.md` の REQ-ID に紐付ける（トレーサビリティ）
- セキュリティ関連タスク（認証・暗号化等）は Phase 1 に配置する
- 完了基準は「実装した」ではなく「〇〇テストが通った」と具体的に記載する
- 依存関係を明示する（`依存: TASK-001` または `依存: なし`）
