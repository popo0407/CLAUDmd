# Instructions

## 前提
- `documents/spec/requirements.md` が存在しない・未承認の場合は **中断** せよ。
- **コードを書かない。** design.md の完成とユーザー承認が先。
- 出力先: `documents/spec/design.md`

## Step 1: 要件と制約の読み込み

以下をすべて読み込む：
- `documents/spec/requirements.md`
- `documents/spec/tech.md`
- `documents/spec/product.md`（制約・成功基準の確認）

## Step 2: 技術方針の比較検討

各主要コンポーネントについて、**複数の実装方式を比較**する。

比較観点：
1. **コスト**（初期コスト・ランニングコスト）
2. **メンテナンス性**（非エンジニアが維持できるか）
3. **スケーラビリティ**（将来の拡張余地）
4. **既存技術との整合性**（tech.md の技術スタックとの親和性）

比較結果を表形式で提示し、推奨案を1つ示す。
**推奨案を決定するのはユーザー。** AIは提案のみ行う。

## Step 3: design.md の生成（ADR形式）

以下のフォーマットに従って生成する（上限: **150行**）：

```
## 1. アーキテクチャ概要
システム全体の構成（テキストまたは箇条書き）

## 2. コンポーネント設計

### [コンポーネント名]
- 役割: 〜
- 入力: 〜
- 出力: 〜
- 処理フロー: 〜

## 3. 意思決定の記録（ADR）

### ADR-001: [決定タイトル]
- 決定日: YYYY-MM-DD
- 状態: 採用済み / 廃止 / 検討中
- 背景: なぜこの決定が必要だったか
- 選択肢:
  | 案 | メリット | デメリット |
  |----|----------|------------|
  | A案（採用） | 〜 | 〜 |
  | B案 | 〜 | 〜 |
- 決定理由: なぜA案を選んだか（ユーザーの要望・制約を明記）
- 影響: この決定によって何が制約されるか

## 4. データフロー
処理の流れをテキストで記述

## 5. セキュリティ考慮事項
個人情報・認証・権限管理等

## 6. 非機能要件への対応
パフォーマンス・可用性・監視等
```

ADRフォーマットルール：
- 技術選定の理由と代替案との比較を**必ず**記録する
- 「あえてこの方式を採用した」理由を明記する（将来のAIが誤って設計を破壊しないための保険）
- 実装中に設計を変更した場合は、その都度 ADR を追記する

## Step 4: ユーザーレビューの要求

```
📐 design.md を生成しました。

レビューをお願いします：
1. ADRの技術選定に納得感はありますか？
2. 設計から漏れている要件はありますか？
3. この設計で予算・期限の制約を満たせますか？

承認後、「タスク作成して」でタスクリストを生成します。
```

## 注意事項
- design.md は **150行以内** に収める
- 各ファイルの先頭に **最終更新日** を記載する（例: `最終更新: 2026-04-22`）
- 不確かな推測は記載しない。判断できない場合は「未定」または「要確認」と明示する
- ADRは必ず「なぜそうしたか」の背景を残す（将来のAIが設計を壊さないための保険）
- ユーザーが選択した方式以外のコードは書かない
- セキュリティ上の懸念（個人情報・認証方式等）は必ず明記する
