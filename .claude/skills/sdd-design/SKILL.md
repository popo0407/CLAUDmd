# Instructions

## 前提
- `documents/spec/requirements.md` が存在しない・未承認の場合は **中断** せよ。
- **コードを書かない。** design.md の完成とユーザー承認が先。
- 出力先: `documents/spec/design.md`

## Step 1: 要件と制約の読み込み

以下をすべて読み込む：
- `documents/spec/requirements.md`
- `documents/spec/tech.md`
- `documents/spec/product.md`（制約・成功基準の確認）

## Step 2: 技術方針の比較検討

各主要コンポーネントについて、**複数の実装方式を比較**する。

比較観点：
1. **コスト**（初期コスト・ランニングコスト）
2. **メンテナンス性**（非エンジニアが維持できるか）
3. **スケーラビリティ**（将来の拡張余地）
4. **既存技術との整合性**（tech.md の技術スタックとの親和性）

比較結果を表形式で提示し、推奨案を1つ示す。
**推奨案を決定するのはユーザー。** AIは提案のみ行う。

## Step 3: design.md の生成（ADR形式）

`.claude/rules/spec-design.md` のフォーマットに従って生成する。

## Step 4: ユーザーレビューの要求

```
📐 design.md を生成しました。

レビューをお願いします：
1. ADRの技術選定に納得感はありますか？
2. 設計から漏れている要件はありますか？
3. この設計で予算・期限の制約を満たせますか？

承認後、「タスク作成して」でタスクリストを生成します。
```

## 注意事項
- design.md は **150行以内** に収める
- ADRは必ず「なぜそうしたか」の背景を残す（将来のAIが設計を壊さないための保険）
- ユーザーが選択した方式以外のコードは書かない
- セキュリティ上の懸念（個人情報・認証方式等）は必ず明記する
