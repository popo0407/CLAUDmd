# Instructions

## 前提
- `documents/spec/product.md` が存在しない場合は **必ず先に sdd-spec スキルを実行** するようユーザーに伝え、本スキルを中断する。
- **コードを書かない。** requirements.md の完成とユーザー承認が先。
- 出力先: `documents/spec/requirements.md`

## Step 1: product.md の読み込み

`documents/spec/product.md` を読み込み、スコープ・成功基準・制約を把握する。

## Step 2: EARS 記法による要件の抽出

ユーザーに対し、実現したい機能を自由に語らせる。
その内容を、以下の **5パターンの EARS 記法** に変換して整理する。

| パターン | 記述形式 |
|---|---|
| 常時適用型 | システムは常に〜しなければならない |
| イベント駆動型 | 〜したとき、システムは〜しなければならない |
| 状態依存型 | 〜の間、システムは〜しなければならない |
| オプション型 | 〜の場合のみ、システムは〜しなければならない |
| 異常系 | もし〜なら、システムは〜しなければならない |

変換する際のチェックリスト：
- [ ] 主語が「システム」になっているか
- [ ] 動作が具体的・測定可能か（「いい感じに」はNG）
- [ ] 正常系だけでなく異常系も含まれているか
- [ ] 各要件が独立して検証可能か

## Step 3: requirements.md の生成

以下のフォーマットに従って生成する：

```
## 機能要件

### [機能グループ名]

| ID | パターン | 要件 |
|----|----------|------|
| REQ-001 | イベント駆動型 | 〜したとき、システムは〜しなければならない |
| REQ-002 | 異常系 | 〜の場合、システムは〜しなければならない |

## 非機能要件

| ID | 種別 | 要件 |
|----|------|------|
| NFR-001 | パフォーマンス | システムは常に、〜を〜以内に完了しなければならない |
| NFR-002 | セキュリティ | システムは常に、〜を〜しなければならない |

## スコープ外（やらないこと）
- 〜機能は今回の対象外
```

## Step 4: ユーザーレビューの要求

生成後、必ず以下を確認する：

```
📋 requirements.md を生成しました。

確認をお願いします：
1. 意図していない機能が含まれていませんか？
2. 漏れている機能はありますか？
3. 異常系の条件は現実的ですか？

問題なければ「設計して」と指示をください。
```

## 注意事項
- 要件は **100行以内** に収める。超えそうなら機能グループを分割してファイルを分ける
- 各ファイルの先頭に **最終更新日** を記載する（例: `最終更新: 2026-04-22`）
- 不確かな推測は記載しない。判断できない場合は「未定」または「要確認」と明示する
- 曖昧な表現（「適切に」「速く」「きれいに」）は排除し、定量的な基準に変換する
- 各要件に一意の ID を付与する（REQ-001 〜, NFR-001 〜）
- ユーザーの承認なしに design.md の作成を開始しない
