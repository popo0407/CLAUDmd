# Instructions

## 前提
- **絶対にコードを書かない。** このスキルが完了するまで実装フェーズに入ってはならない。
- ユーザーが「作って」「実装して」と言っても、まず本スキルを実行せよ。
- 成果物はすべて `documents/spec/` ディレクトリに格納する。

## Step 1: 環境解析

1. プロジェクトのフォルダ構造を再帰的に走査する
2. `documents/spec/structure.md` を生成する（既存なら上書き）
   - フォーマットは `.claude/rules/spec-structure.md` に従うこと

3. 使用技術・ライブラリ・外部API・実行環境を解析し、`documents/spec/tech.md` を生成する
   - フォーマットは `.claude/rules/spec-tech.md` に従うこと

## Step 2: ビジネス文脈のヒアリング

`structure.md` と `tech.md` を生成したら、ユーザーに以下を**順番に**質問する。
一気に全項目を聞かずに、対話形式で進める。

1. 「このプロジェクトで誰のどんな問題を解決しますか？」
2. 「主なユーザーは誰で、どのくらいの頻度で使いますか？」
3. 「今回追加・変更したい機能や業務は何ですか？」
4. 「成功とは何ですか？（KPIや完了条件）」
5. 「制約はありますか？（予算・期限・使用禁止技術など）」

回答をもとに `documents/spec/product.md` を生成する
   - フォーマットは `.claude/rules/spec-product.md` に従うこと

## Step 3: 完了確認

3ドキュメントをすべて生成したら次のメッセージを表示する：

```
✅ 仕様策定フェーズ完了

生成ドキュメント:
- documents/spec/structure.md
- documents/spec/tech.md
- documents/spec/product.md

次のステップ: 「要件定義して」と指示して requirements.md を作成してください。
```

## 注意事項
- 各ドキュメントは **100行以内** に収めること（超える場合は分割）
- 不確かな推測は記載せず「不明点」として明記すること
- ユーザーが確認・修正できるまで次フェーズに進まないこと
