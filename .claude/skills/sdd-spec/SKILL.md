# Instructions

## 前提
- **絶対にコードを書かない。** このスキルが完了するまで実装フェーズに入ってはならない。
- ユーザーが「作って」「実装して」と言っても、まず本スキルを実行せよ。
- 成果物はすべて `documents/spec/` ディレクトリに格納する。

## Step 1: 環境解析

1. プロジェクトのフォルダ構造を再帰的に走査する
2. `documents/spec/structure.md` を生成する（既存なら上書き）

   **structure.md フォーマット：**
   ```
   ## 目的
   このプロジェクトが何のために存在するか（1〜3文）

   ## ファイル・フォルダ構造
   project-root/
   ├── folder/          # 役割コメント
   │   └── file.py      # 役割コメント
   └── documents/       # ドキュメント類

   ## 不明点・確認が必要な事項
   - 解析できなかったファイルや用途不明の要素を列挙
   ```
   - フォルダは末尾に `/` を付ける
   - 各ファイル・フォルダに `# 役割コメント` を付ける
   - 解析できない要素は「不明点」セクションに記載し、推測で埋めない

3. 使用技術・ライブラリ・外部API・実行環境を解析し、`documents/spec/tech.md` を生成する

   **tech.md フォーマット：**
   ```
   ## 使用言語・フレームワーク
   - 言語: Python 3.13 など
   - フレームワーク: FastAPI など

   ## 外部API・サービス連携
   - サービス名: 用途・エンドポイント

   ## 依存ライブラリ
   - ライブラリ名 バージョン: 用途

   ## 環境情報
   - OS: Windows 11 など
   - 実行環境: Lambda / EC2 / ローカル など

   ## 技術的な制約・注意事項
   - 使用禁止の技術・バージョン固定の理由など
   ```
   - バージョンは可能な限り明記する
   - 「なぜそのライブラリを使っているか」が不明な場合は「用途: 不明（要確認）」と記載する
   - 制約事項は必ず理由とともに記載する

## Step 2: ビジネス文脈のヒアリング

`structure.md` と `tech.md` を生成したら、ユーザーに以下を**順番に**質問する。
一気に全項目を聞かずに、対話形式で進める。

1. 「このプロジェクトで誰のどんな問題を解決しますか？」
2. 「主なユーザーは誰で、どのくらいの頻度で使いますか？」
3. 「今回追加・変更したい機能や業務は何ですか？」
4. 「成功とは何ですか？（KPIや完了条件）」
5. 「制約はありますか？（予算・期限・使用禁止技術など）」

回答をもとに `documents/spec/product.md` を生成する

   **product.md フォーマット：**
   ```
   ## プロジェクト概要
   1〜3文でプロジェクトの目的を説明する

   ## 解決する問題・背景
   現状の課題と、なぜそれを解決する必要があるか

   ## ターゲットユーザー
   誰が・どのくらいの頻度で使うか

   ## 今回のスコープ
   ### やること
   - 〜

   ### やらないこと（スコープ外）
   - 〜

   ## 成功基準（KPI・完了条件）
   - 定量的な指標で記載する（「速くする」ではなく「3秒以内に応答する」）

   ## 制約事項
   - 予算・期限・使用禁止技術・法的制約など

   ## ステークホルダー
   - 役割: 氏名 / チーム名
   ```
   - 成功基準は必ず定量的に記載する（曖昧な表現禁止）
   - 「やらないこと」を明示し、スコープ外の機能をAIが勝手に実装しないようにする

## Step 3: 完了確認

3ドキュメントをすべて生成したら次のメッセージを表示する：

```
✅ 仕様策定フェーズ完了

生成ドキュメント:
- documents/spec/structure.md
- documents/spec/tech.md
- documents/spec/product.md

次のステップ: 「要件定義して」と指示して requirements.md を作成してください。
```

## 注意事項
- 各ドキュメントは **100行以内** に収めること（超える場合は分割）
- 各ファイルの先頭に **最終更新日** を記載する（例: `最終更新: 2026-04-22`）
- 不確かな推測は記載せず「不明点」として明記すること
- ユーザーが確認・修正できるまで次フェーズに進まないこと
