# Instructions

## 前提
- `documents/spec/design.md` が存在し、ユーザー承認済みであることを確認する
- `documents/spec/requirements.md` も読み込み、全要件をタスクに反映する
- 出力先: `documents/spec/tasks.md`

## Step 1: 全ドキュメントの読み込み

以下をすべて読み込む：
- `documents/spec/requirements.md`（要件リスト）
- `documents/spec/design.md`（技術設計）
- `documents/spec/tech.md`（技術スタック）

## Step 2: タスクの分解ルール

各タスクは以下の基準を満たすこと：
- **1タスク = 1〜2時間で完了できる粒度**（AIが実行する場合も同様）
- **完了基準が明確**（「実装した」ではなく「〇〇テストが通った」）
- **依存関係が明示**されている
- 各タスクに **テスト/検証手順** が紐付いている

依存関係の記載ルール：
- `依存: なし` または `依存: TASK-001`

## Step 3: tasks.md の生成

以下のフォーマットに従って生成する：

```
## 実装フェーズ概要
- 総タスク数: X件
- 完了条件: requirements.md の全 REQ-XXX / NFR-XXX が検証済みであること

## Phase 1: 基盤構築

### TASK-001: [タスク名]
- 概要: 何をするか（1文）
- 対応要件: REQ-001, REQ-002
- 依存: なし
- 実装内容:
  1. 〜を作成する
  2. 〜を設定する
- 完了基準: 〜が正常に動作することをテストで確認
- テスト方法: `python test_xxx.py` を実行し、全テストがPASSすること

## Phase 2: コア機能実装

### TASK-002: [タスク名]
...

## Phase 3: 統合テスト・最終確認

### TASK-XXX: エンドツーエンドテスト
- 完了基準: 全 REQ-XXX / NFR-XXX がテスト済みかつPASS

## 優先順位マップ
- High（必須）: TASK-001, TASK-002
- Medium（重要）: TASK-003
- Low（あれば良い）: TASK-004
```

## Step 4: 優先順位の確認

生成後、ユーザーに確認する：

```
📋 tasks.md を生成しました（計X件）

確認をお願いします：
1. タスクの優先順位は意図通りですか？
2. 漏れているタスクはありますか？
3. 実装から除外したいタスクはありますか？

承認後、「実装して」で実装フェーズを開始します。
```

## Step 5: 実装フェーズへの引き渡し

ユーザーが承認したら、以下を表示する：

```
✅ タスクリスト承認済み

実装フェーズを開始します。
- 各タスクの完了ごとに進捗を報告します
- テスト結果を確認しながら進めます
- 完了後はウォークスルー（walkthrough.md）を生成します
```

「実装」スキルを呼び出すよう促す。

## 注意事項
- タスクは **優先度の高い順**（基盤 → コア機能 → テスト）で並べる
- 各タスクは **requirements.md の REQ-ID に紐付ける**（トレーサビリティの確保）
- セキュリティ関連タスク（認証・暗号化等）は Phase 1 に配置する
- tasks.md も **100行以内** を目標とする（超える場合はフェーズを分割）
- 各ファイルの先頭に **最終更新日** を記載する（例: `最終更新: 2026-04-22`）
- 不確かな推測は記載しない。判断できない場合は「未定」または「要確認」と明示する
