# Instructions

## 前提
- `documents/spec/tasks.md` が存在し、全タスクが完了済みであること
- `documents/spec/requirements.md`・`documents/spec/design.md` を参照する
- 出力先: `documents/spec/walkthrough.md`

## Step 1: 実装結果の収集

以下の情報をすべて収集する：

1. **完了タスク一覧**（tasks.md の全 TASK-ID と完了ステータス）
2. **テスト結果**（各タスクのテスト方法と PASS / FAIL の結果）
3. **実装中の意思決定ログ**（設計から変更した点・追加判断した点）
4. **最終的なファイル構成**（新規作成・変更されたファイル一覧）

## Step 2: walkthrough.md の生成

以下のフォーマットに従って生成する（**100〜150行以内**）：

```
## 実装サマリー
- 実施日時: YYYY-MM-DD
- 対象機能: （product.md の今回スコープを転記）
- 総タスク数: X件 / 完了: X件 / スキップ: X件

---

## 実装ログ

### TASK-001: [タスク名]
- ステータス: ✅ 完了 / ⚠️ 一部対応 / ❌ スキップ
- 実施内容: 何をどのように実装したか（2〜5文）
- 変更ファイル: `path/to/file.py`
- テスト結果: PASS / FAIL（テストコマンドと出力サマリー）
- 特記事項: 設計書から変更した点・理由（なければ「なし」）

---

## 要件カバレッジ

| 要件ID | 要件概要 | 対応タスク | 検証結果 |
|--------|----------|------------|----------|
| REQ-001 | 〜したとき、〜しなければならない | TASK-001 | ✅ PASS |

全要件カバレッジ: X / X

---

## 設計変更・追加決定の記録（ADR補足）

### 変更点1: [タイトル]
- 背景: 実装中に判明した事実や制約
- 変更内容: design.md から何をどう変えたか
- 理由: なぜ変更が必要だったか
- 影響: どのファイル・要件に影響するか

---

## 最終ファイル構成
（新規作成・変更されたファイルのツリー）

---

## 次のステップ（推奨）
1. design.md に ADR 補足を追記する
2. system_tree.md を最新状態に更新する
```

## Step 3: 完了メッセージの表示

```
✅ ウォークスルー生成完了

documents/spec/walkthrough.md を作成しました。

確認事項:
- 要件カバレッジ: X / X
- 未解決の特記事項: X件
- design.md への ADR 追記が必要: あり / なし

問題がなければ system_tree.md を更新してください。
```

## 注意事項
- テスト結果は「実行した事実」のみ記載し、推測を書かない
- 設計から変えた箇所は必ず理由とともに記録する（後からの修正でAIが誤って戻すことを防ぐ）
- ウォークスルーは **100〜150行以内** に収める。超える場合は要件カバレッジ表を別ファイル（`walkthrough_coverage.md`）に分離する
- 各ファイルの先頭に **最終更新日** を記載する（例: `最終更新: 2026-04-22`）
- 変更時は、何を・なぜ変えたかを **ADR補足** として design.md に追記すること
