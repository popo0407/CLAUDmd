## 機能要件

### 認証・ユーザー管理

| ID | パターン | 要件 |
|----|----------|------|
| REQ-001 | イベント駆動型 | 保護者がメールアドレスとパスワードを入力したとき、システムはCognitoで認証し、JWTトークンを発行しなければならない |
| REQ-002 | 状態依存型 | 未認証の間、システムはカレンダー・資料・チャット画面へのアクセスを拒否し、ログイン画面にリダイレクトしなければならない |
| REQ-003 | イベント駆動型 | 招待リンクを受け取った配偶者がアカウント登録したとき、システムは同じ家族グループに紐付けしなければならない |
| REQ-004 | 常時適用型 | システムは常に、Cognitoリフレッシュトークンの有効期限を30日に設定し、アプリ利用中は自動更新することでユーザーが再ログインを求められないようにしなければならない |
| REQ-005 | イベント駆動型 | 保護者がアプリを再度開いたとき、システムはローカルに保存されたリフレッシュトークンで自動認証し、ログイン画面を表示せずにカレンダー画面を表示しなければならない |

### 資料アップロード

| ID | パターン | 要件 |
|----|----------|------|
| REQ-010 | イベント駆動型 | 保護者がPDF・JPEG・PNGファイルを選択し、対象の子供（複数選択可）を指定してアップロードしたとき、システムはS3に保存し、子供情報を付与したAI解析ジョブをキューに登録しなければならない |
| REQ-010a | 常時適用型 | システムは常に、アップロード画面で子供プロフィール一覧を表示し、対象の子供を1人以上選択しなければアップロードを完了させてはならない |
| REQ-011 | 常時適用型 | システムは常に、1ファイルあたりの最大サイズを20MBに制限しなければならない |
| REQ-012 | イベント駆動型 | アップロードが完了したとき、システムは「解析中」ステータスをUIに即時反映しなければならない |
| REQ-013 | 異常系 | 対応外のファイル形式がアップロードされた場合、システムはアップロードを拒否し「PDF・JPEG・PNGのみ対応しています」とエラーメッセージを表示しなければならない |
| REQ-014 | 異常系 | ファイルサイズが20MBを超えた場合、システムはアップロードを拒否し「20MB以下のファイルを選択してください」とエラーメッセージを表示しなければならない |

### AI解析・スケジュール抽出

| ID | パターン | 要件 |
|----|----------|------|
| REQ-020 | イベント駆動型 | S3へのアップロードが完了したとき、システムはLambdaを非同期起動し、Bedrock (jp.amazon.nova-2-lite-v1:0) に対象子供の氏名・学年・クラス情報をコンテキストとして付加した上でOCR・内容解析・スケジュール抽出を実行しなければならない |
| REQ-021 | イベント駆動型 | AI解析が完了したとき、システムは抽出した日程・持ち物・お知らせ・イベント種別（運動会/参観日/遠足/給食なし/その他）を子供IDと紐付けてDynamoDBに保存し、ステータスを「解析完了」に更新しなければならない |
| REQ-022 | イベント駆動型 | AI解析が完了したとき、システムは抽出したイベント（タイトル・日付・時刻・場所・持ち物・子供ID）をカレンダーに自動登録しなければならない |
| REQ-023 | イベント駆動型 | 保護者が解析済み資料を開いたとき、システムはAIが生成した要約（200字以内）と「元の資料を開く」ボタンを表示しなければならない |
| REQ-026 | イベント駆動型 | 保護者が「元の資料を開く」ボタンをタップしたとき、システムはS3署名付きURL（有効期限15分）を発行してブラウザで元のPDF・画像ファイルを表示しなければならない |
| REQ-024 | 異常系 | AI解析がタイムアウト（30秒超）または失敗した場合、システムはステータスを「解析失敗」に更新し、保護者に再試行ボタンを表示しなければならない |
| REQ-025 | 異常系 | PDF内にテキストが存在しない（スキャン画像PDF）場合、システムはNova Liteのビジョン機能で画像として解析し、結果を返さなければならない |

### カレンダー

| ID | パターン | 要件 |
|----|----------|------|
| REQ-030 | 常時適用型 | システムは常に、月表示カレンダーを初期画面として表示しなければならない |
| REQ-034 | イベント駆動型 | 保護者がカレンダー画面で子供フィルターを選択（複数選択可・全員表示も可）したとき、システムは選択した子供のイベントのみに絞り込んで表示しなければならない |
| REQ-035 | 常時適用型 | システムは常に、カレンダー上のイベントを子供ごとに異なる色で区別して表示しなければならない |
| REQ-031 | イベント駆動型 | 保護者がカレンダーの日付をタップしたとき、システムはその日のイベント一覧（タイトル・時刻・持ち物）を表示しなければならない |
| REQ-032 | イベント駆動型 | 保護者がイベントをタップしたとき、システムは元の配布資料のサムネイルと要約を含む詳細画面を表示しなければならない |
| REQ-033 | イベント駆動型 | 保護者が手動でイベントを追加・編集・削除したとき、システムは変更をDynamoDBに保存し、配偶者の画面に30秒以内に反映しなければならない |

### 家族間共有

| ID | パターン | 要件 |
|----|----------|------|
| REQ-040 | 常時適用型 | システムは常に、同じ家族グループに属する夫婦のカレンダー・資料・イベントデータを共有しなければならない |
| REQ-041 | イベント駆動型 | 一方の保護者が資料をアップロードしたとき、システムはもう一方の保護者のカレンダーにも解析結果を反映しなければならない |

### 通知・リマインダー

| ID | パターン | 要件 |
|----|----------|------|
| REQ-050 | イベント駆動型 | 保護者がリマインダーを手動設定（X日前・当日など）したとき、システムはEventBridgeにスケジュールを登録しなければならない |
| REQ-051 | イベント駆動型 | 設定したリマインダー時刻になったとき、システムはPWAプッシュ通知でイベントタイトルと持ち物を送信しなければならない |
| REQ-052 | オプション型 | 保護者がプッシュ通知を許可している場合のみ、システムは通知を送信しなければならない |
| REQ-053 | 異常系 | プッシュ通知の送信に失敗した場合、システムは次回アプリ起動時にアプリ内バナーで未読リマインダーを表示しなければならない |
| REQ-054 | イベント駆動型 | AI解析が完了したとき、システムはイベント種別（例：給食なし／参観日／遠足など）をBedrock で判定し、以下の基準でリマインダー候補を自動生成して保護者に提案しなければならない。提案はワンタップで承認・却下できなければならない。基準例：給食なし→前週土曜8:00「お弁当食材の買い物」＋前日20:00「お弁当の準備」／参観日・発表会→1ヶ月前9:00「勤怠申請」＋前日20:00「持ち物・日程の確認」／遠足→前日20:00「持ち物確認」 |
| REQ-055 | イベント駆動型 | 保護者がリマインダー候補を承認したとき、システムはEventBridgeにスケジュールを登録し「登録しました」とトースト通知を表示しなければならない |

### AIチャット

| ID | パターン | 要件 |
|----|----------|------|
| REQ-060 | イベント駆動型 | 保護者がチャット入力欄に質問を送信したとき、システムは①現在選択中の子供フィルター（子供ID）②質問文からBedrock が抽出した期間・イベント種別キーワード、の2軸でDynamoDBを検索し、ヒットしたイベント・持ち物・資料要約をコンテキストとしてBedrockに渡して回答を生成し、10秒以内に返答しなければならない |
| REQ-060a | 常時適用型 | システムは常に、チャット画面上部に「対象の子供」選択UIを表示し、選択した子供のデータのみを検索対象としなければならない |
| REQ-061 | 常時適用型 | システムは常に、他の家族グループのデータをチャット回答に含めてはならない |
| REQ-062 | 異常系 | チャット回答生成が10秒を超えた場合、システムは「回答を生成中です。しばらくお待ちください」メッセージを継続表示しなければならない |

### 子供プロフィール管理

| ID | パターン | 要件 |
|----|----------|------|
| REQ-070 | イベント駆動型 | 保護者が子供プロフィール（氏名・学年・クラス・学校名・アイコン色）を登録したとき、システムはDynamoDBに保存し、アップロード・カレンダー・チャット画面の子供選択UIに即時反映しなければならない |
| REQ-071 | 常時適用型 | システムは常に、1家族グループあたり最大5人の子供プロフィールを登録できなければならない |
| REQ-072 | イベント駆動型 | 保護者が子供プロフィールを編集したとき、システムはDynamoDBを更新し、既存の紐付き済みイベントにも変更後の情報を反映しなければならない |
| REQ-073 | 異常系 | 子供プロフィールが0件の状態で資料アップロード画面を開いた場合、システムは「先に子供プロフィールを登録してください」と表示し、プロフィール登録画面に誘導しなければならない |
| REQ-074 | イベント駆動型 | 保護者が子供プロフィールを削除したとき、システムはその子供に紐付くイベント・資料データを論理削除し、カレンダーから非表示にしなければならない |

## 非機能要件

| ID | 種別 | 要件 |
|----|------|------|
| NFR-001 | パフォーマンス | システムは常に、ページ初期表示を3秒以内に完了しなければならない（スマホ回線4G想定） |
| NFR-002 | パフォーマンス | アップロードからAI解析完了まで、システムは30秒以内に処理を完了しなければならない |
| NFR-003 | コスト | システムは常に、月間AWSコストを150円（≒$1）以下に維持しなければならない |
| NFR-004 | セキュリティ | システムは常に、S3の資料ファイルへのアクセスを署名付きURL（有効期限15分）経由に限定しなければならない |
| NFR-005 | セキュリティ | システムは常に、APIへのすべてのリクエストに対しCognito JWTトークンを検証しなければならない |
| NFR-006 | セキュリティ | システムは常に、DynamoDBのデータアクセスを家族グループIDでスコープし、他グループのデータを返してはならない |
| NFR-007 | 可用性 | システムは常に、AWSのサーバーレス構成を用いてスケールを自動管理し、単一障害点を持ってはならない |
| NFR-008 | UI/UX | システムは常に、iOS Safari・Android Chrome のスマートフォンブラウザで正常に動作しなければならない |
| NFR-009 | UI/UX | システムは常に、タッチターゲットサイズを44px以上に保ちスマホ操作に最適化しなければならない |

## スコープ外（やらないこと）
- LINE / メール通知
- Google Calendar / Apple Calendar との同期
- クラス全体など3人以上での共有
- 多言語対応（日本語のみ）
- 管理者機能（学校側からの配信）
- WAF（コスト削減のため）
